在 Roo Code 里用 DaoXE

Roo Code 是 VS Code 智能体（Cline 分支），支持任何 OpenAI 兼容供应商。指向 DaoXE，用一个 key 调 GPT、Claude、Gemini、DeepSeek 等 —— 还能给每个模式配不同模型。

更新于 2026-07-20

Roo Code 支持任何 OpenAI 兼容端点，所以切到 DaoXE 只是改 base URL 和 key。Roo 的特色是按模式的模型档位 —— Architect 用顶级模型、Code 用便宜模型，都在一个 key 下。省钱到底靠什么，见最便宜 API 教程。

为什么让 Roo Code 走 DaoXE#

  • 按模式选模型。 给 Code、Ask、Architect 分别配不同的 DaoXE 模型。
  • 一个 key，多家厂商。 无需为 GPT、Claude、Gemini、DeepSeek 或 Kimi 分别开户。
  • 即插即用。 若你已在用 Cline，设置几乎一样 —— Roo 是它的分支。

在 Roo Code 里配置#

  1. 打开 Roo 设置。 点 Roo Code 面板的齿轮，在 Providers 里选 OpenAI Compatible。
  2. 填 Base URL 和 key。 Base URL 填 https://daoxe.com/v1（到 /v1 为止，不带 /chat/completions），粘贴 DaoXE key。
  3. 选 Model ID。 填入 GET /v1/models 里的准确 ID；Roo 可按你的 key 拉取列表。
  4. 按模式分配（可选）。 给 Code、Ask、Architect 配不同模型，平衡成本与质量。

值得注意的坑

Base URL 不要带 /chat/completions —— Roo 会自己补路径。智能体动作需要原生工具调用；选支持工具的模型，否则编辑会失败。模型 ID 与账号绑定（见 GET /v1/models）。来回切换时保留独立的供应商档位。更多客户端：客户端设置说明。

验证你拿到的是真模型#

先证明端点可用，再怀疑客户端 —— 这一步失败，改任何设置都没用：

bash
export DAOXE_API_KEY="your_api_key"

# List the exact model IDs your account can call
curl --fail-with-body --show-error --silent \
  https://daoxe.com/v1/models \
  -H "Authorization: Bearer ${DAOXE_API_KEY}"

确认端点有响应，再在 temperature 0 下用一个难题与官方 API 对比，排除掉包：

bash
curl --fail-with-body --show-error --silent \
  https://daoxe.com/v1/chat/completions \
  -H "Authorization: Bearer ${DAOXE_API_KEY}" \
  -H "Content-Type: application/json" \
  -d '{
    "model": "YOUR_EXACT_MODEL_ID",
    "max_tokens": 64,
    "messages": [{"role": "user", "content": "Say hello in one sentence."}]
  }'

把 开源 benchmark 对准 DaoXE 和官方 API，在 temperature 0 下对比。再学会如何鉴别掉包，让便宜端点无法悄悄把你换成更小的模型。

常见问题#

Roo Code 配置和 Cline 一样吗？

基本一样 —— Roo 是 Cline 分支。选 OpenAI Compatible，填 https://daoxe.com/v1，粘贴 key 和准确模型 ID。

每个模式能用不同模型吗？

能 —— Roo 支持按模式的模型档位：Architect 用顶级模型、Code 用便宜模型。

工具调用失败怎么办？

Roo 需要原生工具调用。换成你账号里支持工具的模型 ID。

在哪找模型 ID？

用 key 调 GET /v1/models；复制准确 ID（与账号绑定）。
