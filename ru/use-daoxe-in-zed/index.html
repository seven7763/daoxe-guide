DaoXE в Zed

Agent в Zed поддерживает кастомных OpenAI-совместимых провайдеров. Добавьте DaoXE в language_models.openai_compatible — и получите GPT, Claude, Gemini, DeepSeek и другие в быстром нативном редакторе, одним ключом.

Обновлено 2026-07-20

Zed позволяет добавить OpenAI-совместимого провайдера из настроек Agent или прямо в settings.json. Укажите api_url на DaoXE и перечислите нужные модели; ключ хранится вне файла (env или UI). Один ключ охватит всех вендоров — см. гайд по дешёвым API.

Зачем направлять Zed через DaoXE#

  • Нативный быстрый редактор. Панель Agent в Zed делает правки вашей моделью DaoXE.
  • Явные модели. Точные ID и контекстные окна в available_models.
  • Один ключ, много вендоров. Без отдельных аккаунтов GPT/Claude/Gemini/DeepSeek.

Настройка в Zed#

  1. Откройте настройки Agent. Выполните agent: open settings и нажмите Add Provider — или правьте settings.json напрямую.
  2. Задайте api_url и модели. В language_models.openai_compatible добавьте провайдера с api_url: https://daoxe.com/v1 и записью в available_models (точный ID + max_tokens).
  3. Безопасно укажите ключ. Введите в UI или задайте env DAOXE_API_KEY — Zed выводит имя переменной из названия провайдера.
  4. Выберите и проверьте. Выберите модель в панели Agent и сделайте небольшую правку.
json
// Zed settings.json  (agent: open settings -> Add Provider, or edit directly)
{
  "language_models": {
    "openai_compatible": {
      "DaoXE": {
        "api_url": "https://daoxe.com/v1",
        "available_models": [
          { "name": "YOUR_EXACT_MODEL_ID", "display_name": "DaoXE model", "max_tokens": 128000 }
        ]
      }
    }
  }
}
// Key is NOT stored here. Set it in the UI, or as env DAOXE_API_KEY
// (Zed derives <PROVIDER_ID_UPPERCASE>_API_KEY from the provider name).

Что важно учесть

Никогда не храните ключ в settings.json — Zed читает переменную <ID_ПРОВАЙДЕРА_ЗАГЛАВНЫМИ>_API_KEY, так что провайдер DaoXE → DAOXE_API_KEY. api_url должен включать /v1. Для reasoning-моделей задайте reasoning_effort и нужные capabilities. ID привязаны к аккаунту — см. GET /v1/models.

Убедитесь, что отдают настоящую модель#

Сначала проверьте эндпоинт, а не клиент — если это не работает, настройки не помогут:

bash
export DAOXE_API_KEY="your_api_key"

# List the exact model IDs your account can call
curl --fail-with-body --show-error --silent \
  https://daoxe.com/v1/models \
  -H "Authorization: Bearer ${DAOXE_API_KEY}"

Проверьте эндпоинт независимо от Zed, затем сравните фиксированный промпт с официальным API при temperature 0:

bash
curl --fail-with-body --show-error --silent \
  https://daoxe.com/v1/chat/completions \
  -H "Authorization: Bearer ${DAOXE_API_KEY}" \
  -H "Content-Type: application/json" \
  -d '{
    "model": "YOUR_EXACT_MODEL_ID",
    "max_tokens": 64,
    "messages": [{"role": "user", "content": "Say hello in one sentence."}]
  }'

Проверяйте нас, а не верьте на слово

Направьте открытый бенчмарк на DaoXE и на официальный API и сравните при temperature 0. Затем изучите, как поймать подмену модели, чтобы дешёвый эндпоинт не подменил вам модель на меньшую.

Частые вопросы#

Как Zed хранит ключ?

Не в settings.json. Используйте UI или задайте DAOXE_API_KEY — Zed выводит имя из названия провайдера (ЗАГЛАВНЫМИ + _API_KEY).

Что писать в api_url?

https://daoxe.com/v1. Должен включать /v1.

Обязательно ли перечислять модели?

Да — добавьте точные ID в available_models с max_tokens (контекст).

Reasoning-модель не думает?

Задайте reasoning_effort и нужные capabilities в записи модели.
